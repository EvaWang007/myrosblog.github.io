## ROS2结构
   本次机器人仿真通过 ROS 操作系统去搭建机器人模型，验证设计的路径规划算法和传感器融合的性能。ROS 操作系统系统从系统实现上通常分为文件系统层、计算图层。文件系统层.主要是实现相关功能的一些功能包，例如程序代码，配置文件等等。计算图层如图 4-2 所示，形象地揭示了其各模块的运行机制：它们通过独立的节点运作，同时以拓扑结构相互连接，从而形成一个错综复杂的系统网络。
   
   🕸️话题通信是 ROS 中使用频率最高的一种通信模式，话题通信是基于发布订阅模式的，即：一个节点去发布相关消息，另一个节点去订阅该消息。以移动机器人导航模块发布运动消息为例，导航模块会根据传感器采集的数据计算出运动控制信息发布给底盘，这就是发布方；同时底盘也会有一个节点订阅运动信息并且最终转化为控制电机运动的脉冲信号，这就是订阅方。🕷️

## ROS2的底层结点运行原理
     就是各个节点的通信，不同的节点会接收到不同信息，它会选择性的发布它所知道的信息。其他节点进行信息处理的时候需要用到其他节点的接收信息，就向所需节点订阅所需要的信息。有一些信息是许多节点都需要公用的，可以写进一个话题。具体解释如下：
     (1)节点：在 ROS 中，节点是运行在计算机上的最小独立实体，它们负责处理数据、执行任务和与其他节点交互。每个节点都有一个唯一的名称，并通过发布消息、接收消息以及提供服务来实现通信。
     (2)消息：在 ROS 系统中，消息被定义为一种特定的数据结构，它由多种类型化的字符构建而成，这些字符不仅支持标准的基元数据类型，例如整数、浮点数、布尔值等，还兼容基元类型的数组结构。
     (3)话题：话题是 ROS 中最常见的通信方式，它是一种单向的数据流，类似于公告板。节点之间可以发布数据到某个主题，其他订阅该主题的节点会接收到这些数据。主题通常是实时的，适合于传感器数据、状态更新等不需要响应的消息传递。
     (4) 服务：服务则是双向通信，节点间通过请求-应答的方式进行交互，定义了固定的接口和输入/输出类型，一个节点发送一个请求到另一个节点，等待对方应答后再继续操作。这种方式适用于需要更复杂交互或者对响应时间有较高要求的操作
